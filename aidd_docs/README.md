# AI-Driven Dev Docs

AIDD structures your AI coding assistant with commands, agents, rules, and memory so it produces consistent, high-quality code.

## How It Works

1. **Memory** (`{{DOCS}}/memory/`) — Your project context. Generated by `/init`, read by the AI on every conversation. Architecture, conventions, testing setup, deployment — all in one place.
2. **Commands** — Slash commands (`/plan`, `/implement`, `/commit`...) that trigger structured prompts. Each command follows a defined workflow.
3. **Agents** — Specialized AI personas for specific tasks (TDD, frontend validation, code quality).
4. **Rules** — Coding standards the AI follows automatically (naming, structure, patterns).
5. **Skills** — Reusable workflows and scripts the AI can load when needed.
6. **Templates** — Scaffolding for agents, commands, rules, skills, and documentation.

## Development Flows

### Quick: `/plan` → `/implement` → `/commit`

Small tasks, clear scope. Plan, implement, commit.

### Standard: `/plan` → `/isolate` → `/implement` + `/assert` + `/commit` → `/review_code` → `/create_request`

Features that need isolation and review. Worktree keeps your main branch clean.

### Full (6 steps)

`/brainstorm` → `/new_issue` → `/plan` → `/isolate` → `/implement` + `/assert` + `/commit` → `/review_code` + `/review_functional` + `/create_request` → `/learn`

For features where requirements are unclear. Starts with clarification, ends with documentation.

## Specific Workflows

| Scenario                    | Commands                                                      | When                                                           |
| --------------------------- | ------------------------------------------------------------- | -------------------------------------------------------------- |
| **New feature**             | `/brainstorm` → `/plan` → `/implement`                        | Requirements need clarification, then plan and build           |
| **Bug fix (unknown cause)** | `/debug` → fix → `/learn`                                     | Root cause analysis: hypotheses, validate, fix                 |
| **Bug fix (reproducible)**  | `/reproduce` → test → fix → PR                                | TDD approach: failing test first, then minimal fix             |
| **Refactoring**             | `/codebase_audit` → `/plan` or `/master_plan` → `/implement`  | Audit first, split into phases if risk ≥ 3                     |
| **TDD**                     | `/kent` agent → `/write` → implement → refactor               | Red-Green-Refactor cycle, one behavior at a time               |
| **Frontend from design**    | `/image_extract_details` → `/implement_from_design` → `/iris` | Extract specs, implement, validate render vs. specs            |
| **Test coverage**           | `/list_untested` → `/write` → validate                        | Target untested behaviors, aim for 80% coverage                |
| **E2E testing**             | `/test_journey`                                               | Validate critical user journeys in browser                     |
| **Code review**             | `/review_code` + `/review_functional`                         | Technical review (rules, patterns) + functional review (specs) |
| **Versioning**              | `/commit` → `/create_request` → `/tag`                        | Conventional commits, PR with template, semantic versioning    |

## Validation Rules

- **Never implement a plan with < 90% confidence.** If the plan is uncertain, go back to `/plan`.
- **`/assert` after every implementation phase.** Tests, types, build, lint must pass.
- **Max 3-5 iterations** before escalating to human. AI loops have guardrails.
- **Separate structural from behavioral changes.** Different commits for refactoring vs. new behavior.

## Context Engineering

The framework is extensible. You can generate new context, rules, agents, commands, and skills to match your project.

### `/init` — Initialize Project Memory

Run once when installing the framework. Analyzes your project and generates memory files in `{{DOCS}}/memory/`:

- `project_brief.md` — Vision, objectives, what the project does
- `architecture.md` — Tech stack, patterns, folder structure
- `codebase_map.md` — Key files, entry points, dependencies
- `coding_assertions.md` — Linting, formatting, type checking commands
- `testing.md` — Test framework, patterns, coverage targets
- `deployment.md` — CI/CD, environments, infrastructure
- `vcs.md` — Branching model, commit conventions

The AI reads these files on every conversation start. Keep them up to date.

### `/generate_rules` — Create Coding Rules

Creates project-specific rules the AI follows automatically when matching files are edited.

Rules are organized by category:

- `00-architecture` — System patterns (Clean, Hexagonal, API design)
- `01-standards` — Style, naming, formatting
- `02-programming-languages` — Language-specific rules
- `03-frameworks-and-libraries` — Framework patterns
- `04-tooling` — Tool and infra configuration
- `05-testing` — Test patterns and conventions
- `06-design-patterns` — Code design patterns
- `07-quality` — Performance and security
- `08-domain` — Business logic rules
- `09-other` — Edge cases

Rules are ultra-short (3-7 words per bullet) and auto-loaded via file glob matching.

### `/generate_agent` — Create Specialized Agents

Creates an agent file with a standard 7-section structure: front-matter, identity, rules, resources, input, instruction steps, output.

Use agents when you need:

- **Behavioral loops** — Agent iterates until task completion (input → execute → check → output)
- **Context isolation** — Agent focuses on specific files (max 5 recommended)
- **Orchestration** — Chain agents for complex workflows

### `/generate_command` — Create Custom Commands

Creates a new slash command with structured prompts. Commands are one-off actions triggered by the user (`/commit`, `/plan`, `/debug`...).

### `/generate_skill` — Create Custom Skills

Creates a skill package that the AI loads automatically when the context matches.

```text
skill-name/
├── SKILL.md              # Main instructions (mandatory)
├── scripts/              # Reusable code (optional)
└── references/           # Additional docs (optional)
```

Create a skill when a pattern repeats 2-3 times with 90%+ identical instructions.

**Skills vs Agents**: an agent defines _who_ does the work (role, scope, behavior). A skill defines _how_ to do the work (workflow, conventions, scripts). Agents can load skills; skills can be used by multiple agents.

### `/learn` — Capture Knowledge

Run after features that bring decisions or new patterns. Analyzes changes and routes knowledge to the right place:

- `{{DOCS}}/memory/` — Project context updates
- Rules — New conventions to enforce
- Skills — Reusable workflows discovered
- `{{DOCS}}/internal/decisions/` — Architecture choices, technical trade-offs
